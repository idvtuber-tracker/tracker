## Problem

Two channels in MAHA5 and Re:Memories orgs were not tracking data:
- **Reynard Blanc**: Had NO database table (channel never went live, so table was never created)
- **Daisy Ignacia Y**: Had a table but data wasn't being recorded (mismatch in channel names)

Root cause: The tracker only created database tables when channels were first detected as live via activities.list(). If a channel was added to CHANNEL_IDS but never went live before the tracker needed to record data, it would have no table, and data would be silently lost.

## Solution

Three new functions implement progressive protection:

### 1. **_preinit_all_channels()** - Prevent Reynard Issue
Pre-initializes database tables for all configured channels at startup by fetching their names from YouTube API. Ensures every channel in CHANNEL_IDS has a table before any streams are detected.

### 2. **_ensure_channel_table()** - Fallback Safety Net  
Creates tables on-demand during stream collection if they're missing. Catches edge cases where pre-initialization didn't run or channels were added mid-run.

### 3. **_log_channel_status()** - Early Detection
Monitors which configured channels have database tables and logs warnings for any that are missing. Catches future incidents before data is lost.

## Changes

- Added `_preinit_all_channels()` function to initialize all channels at startup
- Added `_ensure_channel_table()` function as on-demand safety net
- Added `_log_channel_status()` function for monitoring and early warning
- Updated `run()` to call `_preinit_all_channels()` after `init_db()`
- Updated stream collection to use `_ensure_channel_table()` fallback

## Testing

After deployment, both channels should:
1. Have pre-created database tables
2. Record data when they go live
3. Appear correctly on the dashboard

## Prevents Future Incidents

If a new channel is added to CHANNEL_IDS in the future, it will automatically get a database table on the next tracker restart, preventing this issue from recurring.
